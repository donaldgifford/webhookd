## Lint configuration quirks

`.golangci.yml` is strict (Uber-style) and enables `revive`, `gocyclo` (complexity 15), `gocognit` (30), `funlen` (100 lines / 50 stmts), `nestif` (4), plus full `staticcheck`/`gosec`/`errcheck` including blank-identifier and type-assertion checks. Test files (`_test.go`) and `mock_*.go` files have relaxations — don't add nolint directives on test code for rules already waived there.

`goheader` is configured against `licenses-header.txt`. Every new Go file needs the SPDX two-line header — `make fmt` won't add it for you. Copy the header from any existing file.

Recurring lint gotchas in this repo (all hit during IMPL-0001):

- **`errcheck` with `check-blank: true`** flags `_, _ = w.Write(...)` — you can't just blank-discard. Either handle the error properly or factor into a helper that documents *why* discarding is safe.
- **`gocritic hugeParam`** flags large value receivers. For interface-fixed signatures (`slog.Handler.Handle` takes `slog.Record` by value), use `//nolint:gocritic` with a reason. For our own constructors that take `*Config` by value at the boundary (e.g., `webhook.NewHandler(HandlerConfig, …)`), the same nolint applies — by-value is intentional so callers can build literals.
- **`noctx`** rejects `httptest.NewRequest`, `http.Get`, `http.Post` in tests. Always use `httptest.NewRequestWithContext` and `http.NewRequestWithContext` + `http.DefaultClient.Do`.
- **`contextcheck`** flags shutdown paths that intentionally use `context.WithTimeout(context.Background(), …)` because the parent ctx is signal-cancelled. The cleanest pattern: extract a named helper (`drainServers`, `shutdownTracerProvider`) and put `//nolint:contextcheck` at the call site with the reason.
- **`revive context-as-argument`** — `context.Context` must be the **first** parameter. `httpx.NewServer(ctx, addr, h, cfg)` is correct; `(addr, h, cfg, ctx)` will fail lint.
- **`nakedret`** triggers when named returns + a bare `return` span more than ~5 lines. Just write explicit returns.
- **`gocritic emptyStringTest`** prefers `s == ""` over `len(s) == 0`.
- **`gocritic httpNoBody`** prefers `http.NoBody` over `nil` for empty request bodies.
- **`gocritic exitAfterDefer`** flags `os.Exit` after `defer cancel()`. Wrap the body in a `realMain() int` helper and `os.Exit(realMain())` so deferred cleanup runs.

## Testing patterns that have bitten us

- **Vec metrics with no children render zero lines** on a Prometheus scrape. Tests asserting "metric `foo` is in the exposition" must first trigger at least one observation on the labeled metric, otherwise the test fails despite registration being correct.
- **`hex.DecodeString` is case-insensitive.** A signature test that compares received-header strings byte-for-byte is wrong. The fuzz target `internal/webhook.FuzzSignatureVerify` caught this — compare decoded byte slices, not raw hex strings.
- **`r.PathValue("provider")` is empty in pre-mux middleware.** Go 1.22+ `ServeMux` only populates path values *after* routing. Middleware in the outer chain (e.g., `httpx.RateLimit`) must parse the URL path manually. We use `providerFromPath("/webhook/<provider>")` exactly for this.
- **OTel `BatchSpanProcessor` leaks goroutines.** `goleak.VerifyTestMain` will fail unless either (a) `tp.Shutdown(ctx)` is awaited before the test exits, or (b) `goleak.IgnoreTopFunction("...batchSpanProcessor.processQueue")` is passed. The integration test in `cmd/webhookd/main_test.go` uses option (b).
- **Shutdown contexts must be detached.** `context.WithTimeout(context.Background(), cfg.ShutdownTimeout)` — never derived from the run-loop ctx — so the drain budget survives a SIGTERM-cancelled parent.
- **Provider allow-list gates required env vars.** Phase 2 defaults `WEBHOOK_PROVIDERS=jsm`; loading webhookd without the JSM-required fields fails fast with `config.ErrJSMFieldsRequired`. Tests that don't exercise JSM behavior must call `withBaselineEnv(t)` (sets `WEBHOOK_PROVIDERS=""` + signing secret) to skip the JSM validation. Phase 1 integration tests in `cmd/webhookd/main_test.go` set `WEBHOOK_PROVIDERS=""` directly. The dedicated JSM unit tests use `jsmEnv(t)` to satisfy required fields and exercise the validation.
- **`KUBEBUILDER_ASSETS` must be an absolute path.** envtest's `Environment.Start()` `fork/exec`s `kube-apiserver` and `etcd` from `KUBEBUILDER_ASSETS`. `go test` runs from each package directory, so a relative path under `build/envtest/...` only resolves for tests at the repo root. The Makefile uses `$(abspath ...)` for this — don't drop the wrapper. Symptom if you do: `fork/exec build/envtest/k8s/.../etcd: no such file or directory` from random envtest tests.
- **CRD `metadata.name` field selectors aren't supported by default.** Since K8s 1.30, server-side `metadata.name=foo` field selectors on custom resources require `selectableFields` in the CRD spec. Without it, the apiserver silently returns the unfiltered set. The executor's `waitForSync` uses a namespace-scoped Watch + per-event predicate filtering instead — each test gets its own namespace so cardinality stays at 1.
- **`tools/watch.UntilWithSync` requires bookmark events under client-go v0.35+.** The `WatchListClient` feature gate is default-on starting at K8s 1.35 (matching `client-go` v0.35), and the reflector backing `UntilWithSync` will sit blocked waiting for a streaming-list bookmark that a hand-rolled `cache.ListWatch` can't supply. Symptom: `hasn't received required bookmark event marking the end of initial events stream` warning, watch deadlocks for the full timeout. Workaround: skip the reflector. The executor uses a single `client.WithWatch.Watch()` + manual `select { case ev <- w.ResultChan(): }` loop — simpler, avoids the bookmark dependency, and fine because we have a hard deadline and don't need auto-reconnect.
- **`client.Apply` (sentinel `Patch` type) is deprecated in favor of `client.Client.Apply()`.** The new method takes an `ApplyConfiguration` generated by `applyconfiguration-gen` for client-go-managed types. We don't generate those for the Wiz operator's typed CRs, so the deprecated `client.Patch(obj, client.Apply, client.FieldOwner(...), client.ForceOwnership)` path stays — silenced with `//nolint:staticcheck // SA1019: typed-CR SSA path`. Don't try to "modernize" without first generating ApplyConfigurations for `wizapi/`.
- **`WEBHOOK_CR_SYNC_TIMEOUT` must be strictly less than `WEBHOOK_SHUTDOWN_TIMEOUT`.** Config validation enforces this — equal values trip `ErrSyncTimeoutTooLong`. The motivation is that a SIGTERM mid-sync needs the shutdown timeout to outlast the longest in-flight watch loop. Phase 6 e2e tests use `SHUTDOWN=20s` / `SYNC=10s`; tests with `SYNC=2s` need `SHUTDOWN >= 5s` for safety margin.
- **End-to-end envtest requires materializing a kubeconfig file.** webhookd's startup path only knows about `WEBHOOK_KUBECONFIG`; envtest hands callers a `*rest.Config` directly. Bridge them via `clientcmd.WriteToFile` — copying `Host` + `CAData` + `CertData` + `KeyData` from `*rest.Config` into a `clientcmdapi.Config` literal — and point `WEBHOOK_KUBECONFIG` at the resulting tempfile. Anything that exercises the full `run(ctx)` boot needs this.

## Architectural patterns

- **Narrow config structs at package boundaries.** Each `internal/` package takes the minimum it needs (`webhook.HandlerConfig`, `httpx.AdminConfig`, `httpx.RateLimitConfig`, future `webhook.ExecutorConfig`, `jsm.Config`) — never the full `*config.Config`. Keeps coupling tight and tests easy to fake.
- **Per-key lazy resources via `sync.Map`.** The rate limiter's per-provider `*rate.Limiter` is the canonical example: `Load` first, `LoadOrStore` on miss, no GC of entries (provider set is bounded by `WEBHOOK_PROVIDERS` allow-list).
- **Single private Prometheus registry.** `observability.NewMetrics` returns a fresh `*prometheus.Registry` every call — no `DefaultRegisterer` — so tests can spin up isolated harnesses without leaking state. The constructor takes `*config.Config` only because `BuildInfo` lives there.
- **Provider parses, executor side-effects (Phase 2).** Providers (`internal/webhook/jsm`, future `slack`, etc.) implement `Provider.Handle` as a *pure* function: bytes in → typed `Action` out, no I/O. The dispatcher hands the action to the executor, which owns every K8s call and span (`k8s.apply`, `k8s.watch_cr`). This split keeps providers trivially unit-testable (no envtest in provider tests; just assert on the returned action) and concentrates cross-cutting concerns — span creation, retry classification, response shaping — in one place. New providers must follow this rule; reaching for `client.Client` from inside `Provider.Handle` is a code smell to push back on.
- **Local stub for cross-repo Go types.** When the project depends on Go types owned by another repo that hasn't published yet (Phase 2's `github.com/donaldgifford/wiz-operator/api/v1alpha1`), ship a local stub at `internal/webhook/wizapi/` mirroring the YAML samples in `docs/examples/samples/`. Everything imports through one alias; the swap to the real module is one line plus stub deletion. Don't generate from CRDs (codegen tooling pulls in `controller-tools` for a tiny win).
- **Single shared K8s scheme + dual client.** `internal/k8s.Scheme` is a package-level `runtime.Scheme` initialized at package init; every package that needs typed K8s access imports `internal/k8s` rather than constructing its own scheme. `k8s.NewClients(cfg)` returns a controller-runtime typed `client.WithWatch` (typed Patch / Get / List / Watch — Phase 4 wires the executor straight off this) plus a `kubernetes.Interface` clientset for any future core-K8s-only paths (events, namespaces) that don't go through CRDs. Both share one `*rest.Config`. Startup-time sanity check: `cfg.CR.APIGroup` must equal `wizapi.GroupVersion.Group`, otherwise typed Patch silently sends to the imported group and never hits the operator.
- **Pin K8s deps to controller-runtime's go.mod.** controller-runtime v0.23.3 requires k8s.io/{api,apimachinery,client-go,...} = v0.35.0; `go mod tidy` will jump to v0.36.0 and break with `HasSyncedChecker` interface mismatches. After any controller-runtime bump, verify the dependent k8s.io modules align — re-pin via `go get k8s.io/api@vX.Y.Z k8s.io/apimachinery@vX.Y.Z k8s.io/client-go@vX.Y.Z`.
- **Watch step is binary; classify only at the apply step.** Apply-step K8s errors (`IsForbidden`, `IsInvalid`, etc.) are deterministic and safe to map onto HTTP status codes. Watch-step `Ready=False` from an operator that talks to a "dumb" upstream API (Wiz here) cannot reliably distinguish "permanent failure" from "upstream had a bad day" — so don't try. Treat non-`Ready=True` as still-pending until the timeout deadline (504); JSM retries handle the rest. If real terminal classification is needed, do it via pre-validation (look up referenced CRs *before* applying) — explicit, not inferred from condition reasons.

## Smoke testing the binary

Until a docker-compose dev stack lands, the quickest smoke test is `docker buildx bake webhookd-local` followed by `docker run -p 8080:8080 -p 9090:9090 -e WEBHOOK_SIGNING_SECRET=topsecret -e WEBHOOK_TRACING_ENABLED=false webhookd:dev`, then curl the admin endpoints and post a signed payload (see the README quick-start for the exact `openssl dgst` command). Disabling tracing avoids a hanging exporter when no OTLP collector is reachable.

## Documentation workflow

`docz` is the documentation CLI. Config is `.docz.yaml`; docs live in `docs/{adr,rfc,design,impl,plan,investigation}`. When adding a new design/decision doc, prefer `docz create <type>` (via the `docz:create` skill) over hand-rolling — it auto-updates the README index table. Run `docz update` after editing frontmatter to refresh indexes.

**Open Questions → Resolved Decisions pattern.** IMPL docs (and to a lesser extent DESIGN docs) ship as Drafts with an Open Questions section. The review conversation — usually a question-by-question walk-through with the user — produces answers; flip the section name to "Resolved Decisions" and rewrite each entry as the answer plus the reasoning. The reasoning matters: future readers shouldn't have to re-derive why we picked option B over A. Cascading consequences from a decision (e.g., a CRD shape change rippling through config field names, extractor logic, and metric labels) get applied to the affected sections in the same pass; don't leave the doc with a Resolved Decision that contradicts the body. Cross-doc follow-ups (a draft DESIGN doc still using a superseded shape) belong in a small "Cross-doc follow-ups" subsection — surfaced explicitly, not buried.
