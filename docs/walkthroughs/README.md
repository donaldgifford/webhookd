# Walkthroughs

Hands-on tutorials for the patterns this repo uses, built up from zero. Each
walkthrough is self-contained — copy-paste the code, follow the steps, end with
a working artifact you can run.

These are **not** RFCs, ADRs, or design docs. They are teaching material: strip
away the production concerns the real `webhookd` codebase carries
(observability, signing, rate limiting, RBAC, ADRs, etc.) and surface the core
mechanics underneath.

| #   | Title                                                                                        | What you'll have at the end                                                                                                                                                                   |
| --- | -------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 01  | [Webhook → Kubernetes CR API from scratch](01-webhook-to-cr-from-scratch.md)                 | A ~150-line stdlib Go service, in a distroless container built with `docker buildx bake`, that creates a CR on every webhook hit (using the **dynamic client** + `unstructured.Unstructured`) |
| 02  | [Typed clients via kubebuilder](02-typed-clients-with-kubebuilder.md)                        | The same service refactored against typed Go structs generated by `kubebuilder` — drops `unstructured`, gains compile-time field safety. Mirrors how this repo talks to the wiz-operator.     |
| 03  | [Productionize: observability + structured logging + graceful shutdown](03-productionize.md) | The same service plus `log/slog`, OTel, Prometheus, Grafana, Tempo, proper error handling, and graceful shutdown — all running locally via `docker compose`                                   |

## When to read these

- You're new to the project and want to understand the _shape_ of what
  `webhookd` does before reading 6,000 lines of production code.
- You want a reference for "how do I write a tiny K8s-CR-creating service
  again?" without dragging in this repo's full toolchain.
- You're explaining the pattern to a teammate and need a single doc to point
  them at.

## When to **not** read these

- You're looking for production guidance — read `docs/design/` and `docs/adr/`
  instead.
- You're debugging this codebase — read `CLAUDE.md` and the `docs/impl/` for the
  shipped patterns.
